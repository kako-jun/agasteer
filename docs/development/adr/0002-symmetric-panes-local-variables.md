# ADR-0002: 左右ペインの完全対称設計（v5・ローカル変数化）

- **日付**: 2025-11-24（CHANGELOG.md記載の実施日）（遡及記録: #228対応、2026-07-16）
- **ステータス**: Accepted

## コンテキスト

2ペイン表示（左右独立ナビゲーション）を導入した際、左ペインは`currentView`/`currentNote`/`currentLeaf`等のグローバルストアで管理し、右ペインは個別のローカル変数で管理するという非対称な実装になっていた。

この非対称設計により、Pull後に左ペインだけ状態が失われる（右ペインは無事）というバグが発生した。原因を調べると、ナビゲーション・CRUD関連の関数が「左ペイン用」「右ペイン用」で別々に実装されており、片方だけ修正が反映されて片方が古いままになる、という同種の不整合が複数箇所に潜んでいた。

## 決定

**左右ペインを完全に対等な実装にする。「コードに差があればバグ」という原則を採用する。**

- `currentView`/`currentNote`/`currentLeaf`、および`subNotes`/`currentNoteLeaves`等の派生ストアを削除する
- 左右とも`leftNote`/`rightNote`、`leftLeaf`/`rightLeaf`等の**ローカル変数**で状態を管理する
- `type Pane = 'left' | 'right'`型を導入し、`goHome(pane)`/`selectNote(note, pane)`/`selectLeaf(leaf, pane)`など全てのナビゲーション・CRUD関数に`pane`パラメータを追加して統一する
- `getBreadcrumbsRight()`のような重複関数を削除し、`getBreadcrumbs(pane)`のような単一関数に統合する
- Pull後の状態復元は「初回Pullだけ特別扱い」をやめ、常にURLから両ペインの状態を復元する
- コメント上の左ペイン中心の表現（"currentNoteが変わるたびに"等）も中立化する

## 却下した代替案

- **右ペインもグローバルストア化する**: 対称性は得られるが、左右2セットのストア（`leftCurrentView`/`rightCurrentView`等）が必要になり、むしろストアの数が倍増して複雑化する。ローカル変数化の方が「ペイン間の独立性」という実態に即しており、シンプルだった。
- **バグ箇所だけの個別修正**: 発覚した「Pull後に左ペインのみ状態が失われる」バグをその場しのぎで直すことも可能だったが、非対称設計自体が同種のバグを繰り返し生む構造的原因であるため、根本の設計を直すことを選んだ。

## 結果

- 8個以上の重複関数を統一し、`App.svelte`だけで約100行のコード削減を達成した（CHANGELOG.md「2025-11-24 - 左右対称設計への大規模リファクタリング」参照）
- `src/lib/stores.ts`（当時）から5個のストアを削除
- 以後、新機能を実装する際は「左右対称かどうか」を設計時に必ず問うルールが定着した（`pane: Pane`パラメータを取る、というシグネチャが以後の標準パターンになった）
- ペインをまたぐ状態同期（例: 検索結果からの別ペインへの遷移、#132/#133）でも、この対称設計が前提として維持されている
